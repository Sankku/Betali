# Betali — Testing Patterns

| Layer | Framework | Config |
|---|---|---|
| Backend unit/integration | Jest 29 + Supertest | `backend/jest.config.js` |
| Frontend E2E | Playwright | `frontend/playwright.config.ts` |

## Backend Testing (Jest)

### Configuration
File: `backend/jest.config.js`

- `testEnvironment: 'node'`
- Test discovery: `tests/**/*.test.js` and `tests/**/*.spec.js`
- Setup file: `tests/setup.js` (runs before every test suite)
- Coverage collected from: `controllers/**/*.js`, `services/**/*.js`, `repositories/**/*.js`, `middleware/**/*.js`
- Coverage output: `backend/coverage/` (text, lcov, html reporters)
- `maxWorkers: 1` — tests run sequentially to avoid database conflicts
- `clearMocks: true` — mocks reset between tests automatically
- `testTimeout: 15000`

### What is Tested

**Unit tests** (`tests/unit/`):
- Service classes in isolation — full business logic branches including validation, not-found, access-denied cases
- Middleware (auth, validation) with fully mocked Supabase and logger
- Repository classes with mocked Supabase client chains
- Validation helpers

**Integration tests** (`tests/integration/`):
- Full workflows that cross service and repository boundaries (e.g., order create → reserve stock → fulfill → cancel)
- Multi-tenant data isolation — verifies Org A cannot read Org B data
- These tests connect to a real Supabase instance using `SUPABASE_SERVICE_KEY`

**API-level tests** (`tests/auth.test.js`, `tests/products.test.js`):
- Call the live running server at `http://localhost:4000` via HTTP
- Use `global.testUtils.makeRequest` helper to send requests and normalize responses
- Verify HTTP status codes and response shape

**Generated tests** (`tests/generated/`):
- Auto-generated by AI tooling (`scripts/ai-test-gen.js`)
- Use Supertest with an inline Express app — no real server needed
- Mock auth middleware inline: `req.user = { id: 'test-user', currentOrganizationId: 'test-org' }`

### Mocking Approaches

**Service dependencies** — constructor injection enables direct mock injection:
```js
mockProductRepository = {
  findByOrganizationId: jest.fn(),
  findById: jest.fn(),
  create: jest.fn(),
  // ...
};
productService = new ProductService(mockProductRepository, mockLogger);
```

**Logger** — mocked via `jest.mock`:
```js
jest.mock('../../utils/Logger', () => ({
  Logger: jest.fn().mockImplementation(() => ({
    info: jest.fn(),
    error: jest.fn(),
    warn: jest.fn(),
    debug: jest.fn()
  }))
}));
```

**Supabase client** — mocked via module mock:
```js
const mockSupabase = { auth: { getUser: jest.fn() }, from: jest.fn() };
jest.mock('../../../lib/supabaseClient', () => mockSupabase);
```

**HTTP-level mocking** (generated tests) — Supertest + inline Express:
```js
app = express();
app.use(express.json());
const mockAuth = (req, res, next) => {
  req.user = { id: 'test-user', currentOrganizationId: 'test-org' };
  next();
};
app.post('/api/orders', mockAuth, (req, res, next) => orderController.createOrder(req, res, next));
```

### Global Test Setup (`tests/setup.js`)
- Defines `global.testUtils` with:
  - `generateTestUser()` — unique email/id per run
  - `generateTestOrganization(ownerUserId)` — unique slug per run
  - `generateTestProduct(organizationId)` — test product fixture
  - `makeRequest(method, endpoint, data, headers)` — axios wrapper returning `{ success, data, status, error }`
  - `cleanupTestData()` — deletes rows matching `%@betali.test` emails and `test-org-%` slugs
  - `waitForServer(maxAttempts, delay)` — polls `/health` until server is ready
- `beforeAll`: waits for server via `waitForServer()`
- `afterEach`: calls `cleanupTestData()` to remove test data after each test

### Test Naming Conventions (Backend)
- File suffix: `.test.js` (Jest default) or `.unit.test.js` / `.integration.test.js` for explicit categorization
- `describe` blocks group by class or feature: `describe('ProductService Unit Tests', () => { ... })`
- Nested `describe` by method name: `describe('createProduct', () => { ... })`
- `test` / `it` names are full sentences starting with "should":
  - `test('should fetch products for organization successfully', ...)`
  - `test('should throw error for duplicate batch number', ...)`
  - `it('should create order and return 201', ...)`

## Frontend E2E Testing (Playwright)

### Configuration
File: `frontend/playwright.config.ts`

- `testDir: './tests/e2e'`
- `baseURL: 'http://localhost:3000'`
- `fullyParallel: true` (sequential on CI with `workers: 1`)
- `retries: 2` on CI, `0` locally
- `reporter: 'html'` → `frontend/playwright-report/`
- Trace: `on-first-retry`; screenshot: `only-on-failure`; video: `retain-on-failure`
- Browser: Chromium only (Firefox and WebKit commented out)
- `webServer` auto-starts `npm run dev` and waits for port 3000

### E2E Test File Structure
```
frontend/tests/
  e2e/
    auth/
      login.spec.ts
      signup.spec.ts
    dashboard/
      dashboard.spec.ts
    multi-tenant/
      organization-isolation.spec.ts
    orders/
      create-order.spec.ts
    products/
      create-product.spec.ts
    warehouse/
      warehouse.spec.ts
  helpers/
    auth.ts          — AuthHelper class (signup, login, logout, token utilities)
    fixtures.ts      — Extended Playwright test with authHelper fixture
    testData.ts      — Static test data and generator functions
```

### What is Tested (E2E)
- Authentication flow: login, signup, redirect guards, validation errors
- Protected route redirect (unauthenticated user → `/login`)
- Core CRUD flows: create order, create product, warehouse operations
- Dashboard rendering
- Multi-tenant data isolation at the UI level

### Fixtures and Helpers

**`fixtures.ts`** — extends Playwright's `test` with custom `authHelper` fixture:
```ts
export const test = base.extend<BetaliFixtures>({
  authHelper: async ({ page }, use) => {
    const authHelper = new AuthHelper(page);
    await use(authHelper);
  },
});
```

**`AuthHelper` class** (`helpers/auth.ts`):
- `signup(email, password, firstName, lastName, organizationName)` — fills form, awaits redirect
- `login(email, password)` — fills form, awaits `/dashboard`
- `logout()` — clicks `[data-testid="user-menu"]` then `[data-testid="logout-button"]`
- `isAuthenticated()` — checks URL does not include `/login` or `/register`
- `getAuthToken()` — reads `sb-auth-token` from `localStorage`
- `setAuthToken(token)` — injects token into `localStorage` for faster test setup

**`testData.ts`**:
- Static credentials: `testData.users.admin` / `testData.users.user`
- Product, warehouse, order fixture objects
- Generator functions: `generateUniqueEmail(prefix)`, `generateUniqueSKU(prefix)`, `generateUniqueOrgName(prefix)` using `Date.now()`

### E2E Test Naming Conventions
- File suffix: `.spec.ts`
- `test.describe('Feature Name', () => { ... })`
- `test('should do something', async ({ page, authHelper }) => { ... })`
- Language-agnostic selectors used for i18n tolerance:
  ```ts
  text=/invalid|incorrect|failed|inválid|incorrecta/i
  ```
- Data-testid attributes used for stable selectors: `[data-testid="user-menu"]`
- Fallback locator chains for resilience: `#create-order-button, button:has-text("New Order")`

### E2E Setup Pattern
Tests that require authentication use `test.beforeEach`:
```ts
test.beforeEach(async ({ page, authHelper }) => {
  await page.addInitScript(() => {
    window.localStorage.setItem('betali_onboarding_completed', 'true');
  });
  await authHelper.login(testData.users.admin.email, testData.users.admin.password);
  await expect(page).toHaveURL(/.*dashboard/, { timeout: 20000 });
});
```

## Test Scripts

### Backend
From `backend/package.json`:
- `bun run test` — runs all Jest tests with coverage
- `bun run test:watch` — watch mode
- `bun run test:coverage` — explicit coverage run
- `bun run test:endpoints` — `scripts/test-endpoints.js` (manual API smoke test)
- `bun run test:api` — `scripts/test-api.js`
- `bun run test:quick` — `scripts/quick-test.sh`
- `bun run test:container` — `scripts/test-container.js`

### Frontend
From `frontend/package.json`:
- `bun run test:e2e` — headless Playwright run
- `bun run test:e2e:ui` — Playwright UI mode
- `bun run test:e2e:headed` — headed browser
- `bun run test:e2e:debug` — debug mode
- `bun run test:e2e:report` — open HTML report

## Current Test Coverage State

### Coverage Status
- **Backend**: Jest coverage configured; collected from `controllers/`, `services/`, `repositories/`, `middleware/`; output at `backend/coverage/`
- **Frontend**: No unit/component tests — Playwright E2E only
- **Frontend components/hooks**: Zero coverage from automated tests

### Test Quality Notes
- Unit tests for services are well-structured with complete happy-path and error-path coverage
- Integration tests use a real Supabase connection and rely on live data; they run sequentially
- `tests/generated/` contains AI-generated tests — quality varies, `.bak` files indicate iteration
- Integration tests in `tests/integration/` include their own colored console logging utilities rather than using Jest matchers exclusively (indicating they were written as standalone scripts and later adapted)
- `tests/setup.js` waits for the server with `waitForServer()` — tests fail if the backend isn't running
- No snapshot testing used anywhere
- No mock service worker (MSW) in frontend — no component-level testing infrastructure exists
